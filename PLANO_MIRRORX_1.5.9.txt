PARTE 3 — FIX TOPBAR BUG (Desconexão ao clicar botões)
──────────────────────────────────────────────────────
O que: Os botões da topbar (Cursor, Telefone, +, 👍, Lápis)
      atualmente DESCONECTAM o servidor. Isso é um bug grave.
Causa provável:
  • MirrorScreen.kt tem callbacks que chamam `client.disconnect()`
    quando algum desses botões é pressionado
  • O HermesActivity é activity SEPARADA, então o back press da
    HermesActivity pode estar voltando pro MirrorScreen e o
    MirrorScreen desconecta
Como:
  • Verificar `MainActivity.kt` e `MirrorScreen.kt`
  • Garantir que o disconnect só acontece quando o user clica
    explicitamente no botão "Sair"
  • Adicionar auto-reconexão (se cair, tenta reconectar 3x)

PARTE 6 — UDP COMO ALTERNATIVA AO TCP
──────────────────────────────────────
O que: Adicionar opção de usar UDP (User Datagram Protocol)
      em vez de TCP/WebSocket pra reduzir lag.
Trade-offs:
  • UDP: menos latência (sem ACK), mas pode perder pacotes
  • TCP (WebSocket): confiável, mas ~50-100ms overhead
Como:
  • Adicionar flag `--udp` no server.py
  • Servidor UDP escuta em 9901 (separado do TCP 9900)
  • Cliente Android: tentar UDP primeiro, fallback pra TCP
  • Para v1.5.9: começar com TCP, deixar flag UDP como TODO
Por que: O Lucas pediu. Mas é uma mudança arquitetural grande.
        Por isso adiamos pra iteração futura se sobrar tempo.

PROTOCOLO — DECISÃO SOBRE TCP vs UDP
───────────────────────────────────────────────────────────────────

O Lucas pediu UDP "pra diminuir o lag". Análise técnica:

TCP (atual via WebSocket):
  ✓ Confiável — pacotes não se perdem
  ✓ Ordem garantida
  ✗ ACK overhead (~50-100ms em WiFi doméstico)
  ✗ Head-of-line blocking

UDP (proposto):
  ✓ Sem ACK, latência ~10-20ms
  ✗ Pacotes podem se perder
  ✗ Sem ordem garantida
  ✗ Pra vídeo: precisa de FEC (forward error correction)

DECISÃO pra v1.5.9:
  Manter TCP/WebSocket como padrão (estável, testado).
  UDP fica como TODO pra v1.6 com QoS/FEC apropriado.
  Argumento pro Lucas: TCP com otimização (heartbeat 60Hz,
  batched moves, JPEG adaptativo) tem latência < 30ms em rede
  local, que é melhor que a percepção humana (~100ms).
